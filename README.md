# Sprintly

Sprintly — SaaS-платформа для управления проектами и внутренними процессами команды.
Система объединяет модуль задач (Kanban), базу знаний и аналитический блок (CRM) в рамках единой архитектуры с ролевой моделью доступа и журналом активности.

Проект ориентирован на демонстрацию production-подхода: изоляция данных по организациям, RBAC на backend, refresh-token rotation, разделение server/client state и продуманная доменная модель.

---

## Архитектура

Система разделена на frontend и backend с независимыми зонами ответственности.

### Frontend

* Next.js (App Router)
* TypeScript
* TanStack Query (server state)
* Redux Toolkit (глобальное состояние и auth)
* Material UI
* Feature-Sliced Design (FSD)
* dnd-kit (drag & drop)
* SSR защищённых страниц
* Optimistic updates
* Error boundaries

Ключевые принципы:

* Разделение server state и client state
* Изоляция бизнес-логики на уровне features
* Минимизация проп-дриллинга
* Управление доступом на уровне UI через role-based rendering
* Централизованная обработка 401 и auto-refresh токена

---

### Backend

* NestJS
* Prisma ORM
* PostgreSQL
* JWT (access + refresh)
* Refresh token rotation
* Role-based access control (Guards)
* DTO + class-validator
* Soft delete
* Activity logging

Ключевые принципы:

* Multi-tenant изоляция через organizationId
* Проверка прав доступа на уровне Guard
* Валидация входных данных через DTO
* Явная доменная модель
* Логирование критичных изменений

## Авторизация и безопасность

Реализован production-подход к аутентификации:

* Access token хранится в памяти
* Refresh token хранится в httpOnly cookie
* Реализована ротация refresh токенов
* Автоматический refresh при 401
* Защита API через Guards
* Проверка ролей на backend
* Защита маршрутов на frontend
* Изоляция данных по organizationId

---

## Доменная модель

Основные сущности:

* Organization
* User
* Project
* Task
* Comment
* KnowledgeNote
* ActivityLog

Каждая сущность привязана к организации. Данные между организациями полностью изолированы.

---

## Архитектурные решения

* Multi-tenant модель через organizationId вместо отдельных БД
* Разделение server/client state через TanStack Query + RTK
* RBAC реализован централизованно через Guards
* Soft delete вместо физического удаления для сохранения истории
* Activity Log как отдельная доменная сущность, а не побочный эффект
